# V0.3 & V0.4 Architecture Vision
*Drafted: 2026-03-19*

This document outlines the core architecture for the upcoming v0.3 (Managed Cloud & Automation) and v0.4 (Community Extensions) phases of izhubs ERP, based on enterprise SaaS best practices.

## 1. Managed Cloud & Đa cấu trúc ngành (Multi-Tenant SaaS) - v0.3

**Mục tiêu**: Nâng cấp hệ thống từ Single-tenant (Self-host) sang Multi-tenant hoàn chỉnh để vận hành bản Cloud Hosted ($19/mo).

**Thiết kế Kiến trúc:**
- **Isolation bằng PostgreSQL RLS**: Tái sử dụng tối đa kiến trúc RLS đã thiết lập ở v0.1. App layer không bao giờ được phép `SELECT *` mà không set context `app.current_tenant_id` trong transaction.
- **SaaS Billing Schema**: Triển khai cấu trúc Subscription chuẩn:
  - Bảng `tenant_subscriptions`: `tenant_id`, `stripe_customer_id`, `stripe_sub_id`, `plan` (Free/Pro/Enterprise), `status`, `current_period_end`.
- **Thanh toán (Stripe)**: Gắn Stripe Checkout cho luồng nâng cấp gói, sử dụng Stripe Webhooks (đảm bảo idempotency - chống duplicate event) để tự động cập nhật trạng thái `status` (active, past_due, canceled).
- **Feature Flags**: Quản lý giới hạn tính năng (VD: số lượng Contact tối đa, chặn không cho dùng Automation Engine ở plan Free) thông qua một middleware kiểm tra `tenant_subscriptions.plan`.

## 2. Nền tảng Tự động hóa (Automation Engine) - v0.3

**Mục tiêu**: Hệ thống IFTTT / Zapier nội bộ siêu nhẹ, lưu trữ qua JSON.

**Thiết kế Schema (JSON-first):**
- **Bảng `tenant_automations`**: `id`, `tenant_id`, `name`, `is_active`, `trigger_event` (vd: `deal.won`), và một cột `flow_payload` (JSONB).
- **Cấu trúc `flow_payload`**:
  ```json
  {
    "conditions": [{ "field": "deal.value", "operator": ">", "value": 5000 }],
    "steps": [
      {
        "id": "step_1",
        "action": "core.email.send",
        "mapping": {
          "to": "{{deal.contact_email}}",
          "subject": "Chúc mừng chốt deal",
          "body": "Deal {{deal.name}} trị giá {{deal.value}} đã thành công."
        }
      }
    ]
  }
  ```

**Kiến trúc Thực thi Queue-based:**
- Dùng **Event Bus** hiện có: Khi `deal.won` xảy ra, Event Bus gom data và tìm tất cả `tenant_automations` match với trigger.
- Payload sẽ được ném vào **Redis Queue** (BullMQ) thay vì chạy đồng bộ.
- Một **Worker Node** độc lập sẽ xử lý queue này: parse chuỗi mác-cơ `{{...}}`, sau đó gọi các module Action tương ứng (Gửi email, Post webhook). Nếu lỗi (API timeout), worker sẽ auto retry 3 lần với backoff.

## 3. Hệ sinh thái Mở rộng (Community Extensions & App Store) - v0.4+

**Mục tiêu**: Marketplace phi tập trung. Dev ngoài có thể viết Extension mà không cần commit code vào monorepo của izhubs ERP.

**Kiến trúc Webhook-driven "Ngược" (No-code execution inside ERP):**
- Khác với WordPress phải nhúng mã PHP độc hại, Extension của izhubs sẽ là một Web App dạng Serverless (Vercel/Cloudflare Workers) do Dev tự host.
- **Manifest URL**: Admin cài Extension bằng cách nhập một URL (vd: `https://my-sms-extension.com/manifest.json`).
- ERP sẽ tải Manifest này về chứa:
  - `name`, `permissions_requested` (vd: `read:contacts`).
  - Danh sách **Triggers** mà nó muốn lắng nghe (ERP sẽ thiết lập Webhook tự động bắn payload cho dev khi có event).
  - Danh sách **Actions** mà nó cung cấp ngược lại cho Automation Engine của ERP (Kèm Zod/JSON Schema UI để hiển thị trực tiếp trong ERP).
- **Lưu trữ dữ liệu**: Thay vì extension tự quản lý DB, ERP cung cấp bảng `tenant_module_records` (Cột JSONB `payload`). Extension được cấp 1 API Key chỉ có quyền đọc/ghi vào `namespace` của chính nó trên ERP. Từ đó giúp dữ liệu doanh nghiệp không rời khỏi hệ thống ERP.
